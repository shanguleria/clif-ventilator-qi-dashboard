# Bundle Scorecard — Methods

How the bundle scorecard (`scorecard/build_scorecard.py`) assembles the per-metric tiles, and its
grain-fallback and unit-grouping rules. Prose is hand-written; the enabled-metric list and per-tile grain
in the AUTOGEN block are stamped from `config.json` + the feeds on every rebuild.

> **Reproducibility:** the dashboard produces the same numbers every time it is run, at every site. For a
> plain-English, clinician-focused explanation of how that is guaranteed (and why the unit label is "the
> unit the patient started the day in"), see **[`determinism.md`](determinism.md)** in this folder.

## What the scorecard is
A single HTML page (`output/dashboard/scorecard.html`) showing one tile per QI metric, with global
Week / Month / Unit selectors and a "Group ICUs by" (type vs specific unit) toggle. The combiner is a
**pure aggregator** — it computes nothing clinical; each metric owns its own pipeline and emits a
PHI-free tile feed, and the scorecard just collects and renders them.

## Feed discovery (by convention)
The combiner reads `config.json → metrics` (the enable-list) and, for each metric `<m>`, loads
`metrics/<m>/output/final/tile_feed_<m>.json` (`load_feed`). Each feed is validated: `schema_version == 1`,
required keys present (`schema_version, metric_id, title, grain, headline`), and a PHI guard rejects any
`hospitalization_id`/`patient_id` substring. A missing or invalid feed renders as a "coming soon"
placeholder rather than failing the build. There is **no `scorecard_tiles` registry key** — it was
retired in the monorepo move; discovery is purely by the `metrics` list + path convention.

Each feed's `detail_href` drill-down dashboard is copied into `output/dashboard/` so its "View details →"
link resolves within the shipped bundle.

## Grain fallback (client-side `resolve()`)
The tile a user sees for a given Unit × Period selection is resolved per feed:
- **Unit fallback** — if the feed has no cell for the selected specific unit/type, it falls back to the
  site-wide `__ALL__` cell with a `· site-wide` badge.
- **Week → containing month** — a feed that publishes only monthly grain (proning) resolves a Week pick
  to that week's containing month (ISO-week Thursday → `YYYY-MM`), badged `· Mon YYYY · month`.
- **Drop to all-time** — if neither the selected period nor its fallback exists, the tile shows all-time.

This lets metrics with different natural cadences (LPV/SAT/SBT answer weeks exactly; proning is
month-coarse) coexist on one set of global controls.

## Unit grouping (type vs specific unit)
Each feed carries a `dims` block: `type` (ICU `location_type`, the back-compat default) and `name`
(specific `location_name`), with `parent` nesting each name under its type and optional friendly `labels`
(`config.json → unit_labels`). The "Group ICUs by" toggle switches the whole scorecard between the two
dimensions; feeds without name cells fall back to site-wide.

## Period bucketing & small cells
Period keys are month `"YYYY-MM"` and ISO week `"YYYY-Www"`, bucketed on each metric's own unit-of-
analysis date. Slices below a metric's `reporting.small_cell_min_den` are grayed, not hidden.

## Refresh model
`refresh_scorecard.sh` re-emits the LPV feed, rebuilds the scorecard, and then runs
`docs/build_methods.py` — so a re-render also refreshes these methods docs, with **no CLIF re-read**.
`run_bundle.sh` does the same after the full LPV pipeline. Other metrics (proning/sat/sbt) re-emit their
own feeds by running their own pipelines; the combiner collects whatever feeds currently exist.
